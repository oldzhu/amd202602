# MXFP4-MM 实现说明

[English Version](EXPLAINED.md)

## 这段实现做了什么

核心任务是计算：

- 输入：`A [M, K] bf16`
- 输入：`B [N, K]` 的 MXFP4 量化版本
- 输出：`C [M, N] bf16`

数学上就是 `A @ B.T`，只是 B 使用了 MXFP4 压缩表示。

## 数学含义

普通矩阵乘是：

- `C[i,j] = Σ A[i,k] * B[j,k]`

加入 MXFP4 后，本质上变成：

- 用量化后的 FP4 值加 scale 近似恢复真实值
- 每 32 个元素共享一个 scale

## 数据流

可按以下顺序理解：

1. A 侧激活先量化到 MXFP4
2. A 侧 scale 根据底层 kernel 需要做 shuffle
3. B 侧通常已经由任务预先提供量化与洗牌结果
4. 最后交给 `gemm_a4w4` 这类底层 kernel 执行矩阵乘

## 关键函数说明

### `quantize_to_mxfp4(...)`

- 作用：把 BF16 张量压缩成 MXFP4 数据和对应 scale
- 关键关系：
  - 两个 FP4 值打包成一个字节
  - 每 32 个值使用一个 scale

### `aiter.gemm_a4w4(...)`

- 作用：执行 4-bit 量化矩阵乘
- 典型输入包括：
  - 量化后的 A
  - 预洗牌的 B
  - A/B 两侧的 scale
- 如果 B 已经预洗牌，需要明确告诉 kernel 不要重复洗牌

## 为什么要 Shuffle

- shuffle 的目的是匹配 GPU 更高效的 tile 访问方式
- 不是改变数学结果，而是改变底层内存组织方式
- 这个点非常重要：错误理解 shuffle 常常会导致“看起来 dtype 对了，但结果或性能都不对”

## MXFP4 格式要点

### FP4 值（E2M1）

- 每个值只有 4 bit
- 可表示的数值集合很有限
- 因此必须依赖 block scale 才能覆盖更宽的动态范围

### E8M0 scale

- 只保存指数信息
- 适合用作 power-of-two 缩放
- 和 block quantization 组合后，可以用极低成本表达较宽的值域

## 为什么这种方法快

1. 带宽更低：B 侧存储显著压缩
2. 计算密度更高：低精度格式更适合底层硬件路径
3. 若量化、shuffle、GEMM 能更紧密融合，launch 与数据搬运开销也能下降

## 关键结论

- MXFP4-MM 的核心挑战不是单纯写出矩阵乘公式，而是正确处理量化、scale 与布局契约
- 如果契约对了，底层内核可以非常快
- 如果契约错了，最常见症状就是结果错误、维度不匹配，或者性能大幅退化
